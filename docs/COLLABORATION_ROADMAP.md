# Roadmap: Командные функции LinkLayer

**Статус:** План для будущих версий (не входит в MVP)  
**Целевая версия:** Фаза 3+

---

## Обзор

Этот документ описывает видение развития командных функций LinkLayer для поддержки совместной работы в команде, подобно GitHub для кода, но для управления знаниями.

---

## 1. Целевая аудитория для командных функций

- Небольшие компании (2-50 человек)
- Инди-разработчики и фрилансеры
- Исследовательские группы
- Образовательные учреждения
- Команды проектов

---

## 2. Основные возможности командных функций

### 2.1 Управление доступом

#### Система ролей
```
- Owner (Владелец)
  - Полный контроль над хранилищем
  - Может добавлять/удалять участников
  - Может менять роли участников
  - Может удалить хранилище
  
- Editor (Редактор)
  - Может создавать и редактировать заметки
  - Может создавать связи
  - Может добавлять теги
  - Может участвовать в чате
  - НЕ может управлять участниками
  
- Viewer (Просмотр)
  - Может только просматривать заметки
  - Может просматривать граф
  - Может участвовать в чате
  - НЕ может редактировать
```

#### Приглашение участников
- Приглашение по email
- Генерация ссылки для приглашения
- Автоматическое принятие или требование подтверждения
- История приглашений

### 2.2 Синхронизация в реальном времени

#### Collaborative Editing
- Одновременное редактирование одной заметки несколькими пользователями
- Разрешение конфликтов (Operational Transformation или CRDT)
- Курсоры других пользователей видны в редакторе
- Индикатор "кто сейчас редактирует"

#### Синхронизация графа
- Изменения в графе (новые связи, удаленные узлы) синхронизируются в реальном времени
- Все участники видят актуальное состояние графа

### 2.3 Система комментариев

#### Комментарии к заметкам
- Возможность оставить комментарий к заметке
- Упоминание других пользователей (@username)
- Резолюция комментариев (решено/не решено)
- Уведомления при упоминании

#### Комментарии к связям
- Обсуждение смысла связи между заметками
- Предложение изменений

### 2.4 Встроенный чат

#### Основные функции
- Чат в контексте хранилища
- Возможность прикрепить заметку к сообщению
- Упоминание пользователей
- Поиск по истории чата
- Потоки обсуждения (threads)

#### Интеграция с заметками
- Быстрое создание заметки из сообщения чата
- Ссылка на обсуждение в чате из заметки

### 2.5 Активность и уведомления

#### Лента активности
- Кто создал/отредактировал заметку
- Кто добавил новую связь
- Кто оставил комментарий
- Фильтрация по типу активности

#### Уведомления
- Уведомление при упоминании
- Уведомление при изменении заметки, которую я смотрел
- Уведомление при новом комментарии к моей заметке
- Настройка уведомлений (email, in-app, push)

## 3. Архитектура данных для командных функций

### Новые таблицы

#### VaultMember (расширенная)
```
- id (UUID)
- vault_id (UUID, FK to Vault)
- user_id (UUID, FK to User)
- role (string: "owner", "editor", "viewer")
- joined_at (timestamp)
- invited_by (UUID, FK to User)
- permissions (JSON, для гранулярного контроля)
```

#### Comment (Комментарий)
```
- id (UUID)
- note_id (UUID, FK to Note)
- user_id (UUID, FK to User)
- content (text)
- resolved (boolean)
- created_at (timestamp)
- updated_at (timestamp)
```

#### Activity (Активность)
```
- id (UUID)
- vault_id (UUID, FK to Vault)
- user_id (UUID, FK to User)
- action_type (string: "create_note", "edit_note", "delete_note", "create_link", "delete_link", "comment")
- target_id (UUID, может быть note_id или link_id)
- metadata (JSON)
- created_at (timestamp)
```

#### Notification (Уведомление)
```
- id (UUID)
- user_id (UUID, FK to User)
- type (string: "mention", "edit", "comment", "activity")
- related_vault_id (UUID, FK to Vault)
- related_note_id (UUID, FK to Note, optional)
- read (boolean)
- created_at (timestamp)
```

## 5. Технические решения

### Collaborative Editing
- **Вариант 1**: Operational Transformation (OT)
  - Сложнее в реализации
  - Хорошо изучено
  - Используется в Google Docs
  
- **Вариант 2**: CRDT (Conflict-free Replicated Data Type)
  - Проще в реализации
  - Лучше для offline-first
  - Используется в Figma, Notion
  - Рекомендуется для LinkLayer

### Real-time синхронизация
- WebSocket для real-time обновлений
- Supabase Realtime для синхронизации БД
- Кэширование на клиенте

### Разрешение конфликтов
- Last-write-wins (простой вариант для MVP)
- CRDT для более сложных сценариев
- Явное разрешение конфликтов пользователем

## 6. Примеры использования

### Сценарий 1: Исследовательская группа
1. Профессор создает хранилище "Проект X"
2. Добавляет студентов как editors
3. Студенты создают заметки с исследованиями
4. Обсуждают в чате, прикрепляя заметки
5. Профессор видит активность и комментирует

### Сценарий 2: Инди-разработчики
1. Разработчик A создает хранилище "Идеи для стартапа"
2. Приглашает разработчика B
3. Вместе создают заметки с идеями
4. Соединяют их в граф
5. Обсуждают в чате, как они связаны
6. Экспортируют в документ для инвесторов

### Сценарий 3: Компания
1. Менеджер создает хранилище "Документация проекта"
2. Добавляет разработчиков как editors
3. Добавляет менеджеров как viewers
4. Разработчики документируют архитектуру
5. Менеджеры видят обновления в ленте активности

## 8. Риски и вызовы

### Технические
- Сложность реализации collaborative editing
- Производительность при большом количестве участников
- Синхронизация конфликтов

### Продуктовые
- Конкуренция с Notion, Confluence
- Необходимость обучения пользователей
- Сложность управления правами доступа

### Решения
- Использование CRDT для упрощения
- Оптимизация производительности с самого начала
- Интуитивный UI для управления доступом
- Хорошая документация и примеры
